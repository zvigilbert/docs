The Redwood Cloud Portal uses role-based access control (RBAC) to determine what each user can see and do. Every user account is assigned exactly one role, and that role defines the set of actions the user can perform across the portal — from managing other users to simply viewing environment status. Choosing the right role for each team member keeps your environments secure and ensures people have the access they need without unnecessary privileges.

## Available roles

The portal provides three built-in roles:

- **Administrator** — Full access to all portal features, including user management, SSO configuration, environment settings, and upgrade scheduling.
- **Operator** — Can manage environments and schedule upgrades, but cannot create or manage users or configure SSO.
- **Viewer** — Read-only access. Can view environment status, activity logs, and consumption data, but cannot make changes.

## Permissions matrix

The table below summarizes what each role can do across the main portal sections.

| Feature | Administrator | Operator | Viewer |
|---|:---:|:---:|:---:|
| View environments | Yes | Yes | Yes |
| Connect to environments | Yes | Yes | No |
| Manage environment settings | Yes | Yes | No |
| Schedule upgrades | Yes | Yes | No |
| View upgrade history | Yes | Yes | Yes |
| View activity logs | Yes | Yes | Yes |
| View consumption & contract data | Yes | Yes | Yes |
| Create and manage users | Yes | No | No |
| Assign and change roles | Yes | No | No |
| Configure SSO | Yes | No | No |
| Manage notifications | Yes | Yes | No |

<Note>
  The permissions above reflect portal-level access only. Access to individual Redwood product environments (RunMyJobs, Finance Automation, JSCAPE) is managed within each product, not in the portal.
</Note>

## Assign a role when creating a user

When you [create a new user](/users/managing-users), the **Role** field is required. Select the role that matches what the user needs to accomplish in the portal:

- Assign **Administrator** only to users who are responsible for managing your organization's portal configuration.
- Assign **Operator** to users who manage environments and upgrades on a day-to-day basis.
- Assign **Viewer** to users who need visibility into portal activity without making changes — such as auditors or executives.

## Change a user's role

<Steps>
  <Step title="Open the user's profile">
    In the left navigation sidebar, select **Users & Access**, then select **Users**. Find the user and select their name.
  </Step>
  <Step title="Edit the role">
    Select **Edit**, then choose a new role from the **Role** dropdown.
  </Step>
  <Step title="Save the change">
    Select **Save**. The new role takes effect immediately.
  </Step>
</Steps>

<Warning>
  Only Administrators can change user roles. If you need your own role changed, contact another Administrator in your organization.
</Warning>

## Best practices for role assignment

Following the principle of least privilege — granting users only the access they need — reduces the risk of accidental changes and limits exposure if an account is compromised.

<Tip>
  Start users with the **Viewer** role and elevate their access only when there is a clear need. It is easier to grant additional permissions than to audit the effects of overly broad access.
</Tip>

- **Limit Administrator accounts.** Restrict the Administrator role to a small number of trusted team members who actively manage portal configuration. Having too many Administrators makes it harder to audit changes.
- **Use Operator for day-to-day work.** Team members who connect to environments and manage upgrades but do not need to configure users or SSO should be Operators.
- **Review roles periodically.** As team members change responsibilities or leave the organization, audit your user list to ensure roles still reflect actual needs.
- **Deactivate, don't delete.** When someone no longer needs access, [deactivate their account](/users/managing-users#deactivate-or-remove-a-user) rather than removing it. This preserves their activity history in audit logs.
